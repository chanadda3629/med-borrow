# Medical Equipment Management System - PRD

## Epic 1: Patient Intake & AI-Assisted Assessment
ระบบลงทะเบียนผู้ป่วย ปักหมุดที่อยู่ และประเมินด้วย AI

### Feature Name
Patient Onboarding and AI Clinical Assessment System

### Goal/Why
เพื่อเปลี่ยนผ่านระบบบันทึกข้อมูลผู้ป่วยจากกระดาษมาเป็นดิจิทัล ลดเวลาการทำงานของเจ้าหน้าที่ ป้องกันข้อผิดพลาดในการกรอกข้อมูลที่อยู่ และใช้ AI มาช่วยประเมินความเหมาะสมของอุปกรณ์ลดการพึ่งพาประสบการณ์เฉพาะบุคคลของเจ้าหน้าที่

### User Stories

**User Story 1**
- As a เจ้าหน้าที่ศูนย์ประคองกาย
- I want to กรอกข้อมูลผู้ป่วย, แนพภาพถ่ายสภาพแวดล้อม, เลือกที่อยู่จาก Dropdown พร้อมปักหมุด GPS บน Google Maps และทำแบบประเมินความสามารถทางกายภาพ (Checklist)
- So that ฉันสามารถจัดเก็บข้อมูลผู้ป่วยได้อย่างถูกต้อง แม่นยำ และระบบสามารถส่งข้อมูลนี้ไปให้ AI ประเมินต่อได้ทันที

**User Story 2**
- As a เจ้าหน้าที่ศูนย์ประคองกาย
- I want to เห็นรายการแนะนำอุปกรณ์พร้อมคะแนนความเหมาะสม (%) ที่คำนวณโดย AI
- So that ฉันสามารถใช้เป็นข้อมูลสนับสนุนในการเลือกอนุมัติอุปกรณ์ที่เหมาะสมกับสภาพผู้ป่วยรายนั้นๆ ได้ดีที่สุด

### Functional Requirements

**Form Validation**
- ระบบต้องตรวจสอบเลขบัตรประชาชน (ต้องครบ 13 หลัก และเป็นตัวเลขเท่านั้น)
- ระบบต้องตรวจสอบเบอร์โทรศัพท์ (ต้องครบ 10 หลัก และขึ้นต้นด้วยโครงสร้างเบอร์มือถือที่ถูกต้อง)

**Address Dropdown Cascading**
- ระบบที่อยู่ต้องเป็น Dropdown เลือกทีละขั้น (จังหวัด -> อำเภอ -> ตำบล) เพื่อป้องกัน Human Error
- Auto-populate รหัสไปรษณีย์

**Google Maps Integration**
- รองรับการเปิดแผนที่, ค้นหาที่อยู่, ปักหมุด (Pin Drop)
- บันทึกค่า Latitude, Longitude ลง Database อัตโนมัติ
- Generate Google Maps Link ลง Database อัตโนมัติ

**Media Upload**
- รองรับการอัปโหลดหรือถ่ายภาพ (สภาพแวดล้อมบ้านผู้ป่วย/ตัวผู้ป่วย) ผ่าน Mobile Application

**AI Recommendation Engine Interface**
- ระบบต้องส่ง Payload ข้อมูล (อายุ, เพศ, โรคประจำตัว, ผลจาก Checklist Assessment, รูปภาพ, ระดับความเร่งด่วน) ไปยัง AI Model API
- รับค่า Response กลับมาแสดงผลเป็น list รายการอุปกรณ์ + % Matching Score
- เจ้าหน้าที่คงสิทธิ์ในการกด Override (ตัดสินใจขั้นสุดท้าย)

### Acceptance Criteria

**Scenario 1: สัญญาณ GPS และข้อมูลที่อยู่ครบถ้วน**
- Given: เจ้าหน้าที่อยู่ที่หน้าจอเพิ่มข้อมูลผู้ป่วยใหม่
- When: เจ้าหน้าที่กรอกข้อมูลส่วนตัวครบถ้วน, เลือกที่อยู่จาก Dropdown, ทำการกดปักหมุดบนแผนที่ Google Maps และกดบันทึก
- Then: ระบบต้องบันทึกข้อมูลเข้า Database พร้อมค่าพิกัด Lat/Lng และสร้าง Link Google Maps ได้ถูกต้อง พร้อมเปลี่ยนสถานะไปหน้าถัดไป

**Scenario 2: ข้อมูล Input ไม่ผ่าน Validation**
- Given: เจ้าหน้าที่กรอกเลขบัตรประชาชนไม่ครบ 13 หลัก หรือเบอร์โทรศัพท์ไม่ครบ 10 หลัก
- When: เจ้าหน้าที่กดปุ่ม "ถัดไป" หรือ "บันทึก"
- Then: ระบบต้องไม่บันทึกข้อมูล แสดง Text Error สีแดงเตือนใต้ Field นั้นๆ และ Highlight ช่องที่ผิดพลาด

**Scenario 3: AI ประเมินผลสำเร็จ**
- Given: เจ้าหน้าที่กรอกข้อมูลประเมิน Checklist และอัปโหลดรูปภาพครบถ้วนแล้ว
- When: ระบบส่งข้อมูลไปยัง AI Service สำเร็จ
- Then: หน้าจอต้องแสดงรายการอุปกรณ์ทางการแพทย์เรียงตาม % คะแนนความเหมาะสมจากมากไปน้อย และมีปุ่มให้เจ้าหน้าที่กดยืนยันเลือกอุปกรณ์ตาม AI หรือเลือกอุปกรณ์อื่นเองได้

### Key Constraints/Non-Goals

**Non-Goal สำหรับ Phase 1**
- AI จะยังไม่มีการทำ Image Processing เพื่อวิเคราะห์รูปภาพผู้ป่วยโดยอัตโนมัติ
- รูปภาพใน Phase นี้ใช้เพื่อเป็นข้อมูลให้เจ้าหน้าที่ดูประกอบตาเปล่าเท่านั้น
- AI จะคำนวณจาก Text/Checklist Data เป็นหลัก

## Epic 2: Core Inventory & Borrowing Workflow
ระบบคลังครุภัณฑ์, สถานะ และ Workflow การยืม-คืน

### Feature Name
Real-time Medical Equipment Inventory & Borrowing Workflow Engine

### Goal/Why
เพื่อขจัดปัญหาสต็อกจม, การอนุมัติอุปกรณ์ซ้ำซ้อน (Over-booking), การสูญหายของอุปกรณ์ และช่วยให้เห็นสถานะของครุภัณฑ์ทางการแพทย์ทุกชิ้นแบบ Real-time ตลอดทั้ง Lifecycle

### User Stories

**User Story 1**
- As a เจ้าหน้าที่ศูนย์ประคองกาย
- I want to ตรวจสอบสถานะและจำนวนคงเหลือของอุปกรณ์ในคลังก่อนอนุมัติคำขอ
- So that ฉันจะไม่เผลออนุมัติอุปกรณ์ที่ไม่มีอยู่จริงในคลังให้กับผู้ป่วย

**User Story 2**
- As a เจ้าหน้าที่ศูนย์ประคองกาย
- I want to กดเปลี่ยนสถานะการยืมตามขั้นตอน Workflow (รับคำร้อง -> ประเมิน -> ตรวจคลัง -> อนุมัติ -> จัดส่ง -> รอคืน -> รับคืน) และบันทึกข้อมูลสภาพวัตถุตอนรับคืน
- So that ระบบสามารถเปลี่ยนสถานะเครื่องมือและจำนวนคงเหลือในคลังได้โดยอัตโนมัติ

### Functional Requirements

**Inventory Data Model**
- อุปกรณ์แต่ละชิ้น (Item-level) ต้องมี Unique ID ของตัวเอง
- ประกอบด้วย รหัสอุปกรณ์, เลขครุภัณฑ์, วันที่รับเข้า, และ State/Status ปัจจุบัน

**State Machine (Equipment Status)**
- พร้อมใช้งาน
- ถูกยืม
- รอรับคืน
- ชำรุด
- ซ่อมบำรุง

**Workflow Transition**
- ระบบต้องควบคุม State ของใบคำร้องและการตัดสต็อกตามลำดับที่กำหนด
- ห้ามข้ามขั้นตอน (เช่น จะกด "จัดส่งสำเร็จ" ได้ก็ต่อเมื่อสถานะก่อนหน้าเป็น "เตรียมจัดส่ง")

**Auto-Stock Adjustment**
- เมื่อกด อนุมัติ: ตัดสต็อกอุปกรณ์ชิ้นนั้นๆ ทันที (State เปลี่ยนเป็น ถูกยืม)
- เมื่อกด คืนอุปกรณ์สำเร็จ: สต็อกต้องบวกกลับอัตโนมัติ (State เปลี่ยนเป็น พร้อมใช้งาน หรือ ชำรุด ตามที่เจ้าหน้าที่ติ๊กเลือก)

**Return Management Capture**
- หน้าจอรับคืนต้องบังคับให้ถ่ายรูปภาพอุปกรณ์ ณ วันที่คืน
- เลือกสภาพอุปกรณ์
- ใส่ Text Note ในกรณีที่อุปกรณ์ชำรุดเสียหาย

### Acceptance Criteria

**Scenario 1: อนุมัติอุปกรณ์และตัดสต็อก**
- Given: ใบคำร้องยืม "เตียงผู้ป่วย" อยู่ในสถานะ "ตรวจสอบคลังอุปกรณ์" และมีเตียงเลขครุภัณฑ์ BED-001 สถานะ "พร้อมใช้งาน"
- When: เจ้าหน้าที่กดปุ่ม "อนุมัติ (Approve)" และผูกเลขครุภัณฑ์ BED-001 เข้ากับคำร้องนี้
- Then: สถานะของ BED-001 ในระบบต้องเปลี่ยนเป็น "ถูกยืม" ทันที และจำนวนเตียงที่ "พร้อมใช้งาน" ในคลังรวมต้องลดลง 1 ตัว

**Scenario 2: การรับคืนอุปกรณ์ชำรุด**
- Given: อุปกรณ์เลขครุภัณฑ์ WC-999 อยู่ในสถานะ "ถูกยืม" หรือ "รอรับคืน"
- When: เจ้าหน้าที่ทำรายการรับคืน โดยเลือกสภาพเป็น "ชำรุด" พร้อมอัปโหลดรูปถ่ายความเสียหายและบันทึกข้อมูล
- Then: สถานะของ WC-999 ต้องเปลี่ยนเป็น "ชำรุด" ทันที และจำนวนอุปกรณ์ "พร้อมใช้งาน" ในคลังต้องไม่ถูกบวกเพิ่ม (แต่จำนวนอุปกรณ์ชำรุดในระบบต้องเพิ่มขึ้น 1)

### Key Constraints/Non-Goals

**Key Constraint**
- ระบบจะไม่รองรับการกดอนุมัติยืมแบบไม่มีการระบุเลขครุภัณฑ์ (Serialization)
- ห้ามตัดสต็อกแบบยอดรวม (Bulk) โดยไม่ระบุชิ้นเด็ดขาด

**Non-Goal สำหรับ Phase 1**
- ยังไม่มีระบบ "จองคิวล่วงหน้า (Reservation Queue)"
- หากอุปกรณ์หมดคือหมด ณ ตอนนั้น (ไปทำใน Phase 2)

## Epic 3: Notifications & Analytics Dashboard
ระบบแจ้งเตือนผ่าน LINE และ Dashboard สำหรับผู้บริหาร

### Feature Name
LINE Notification Gateway & Management Dashboard Analytics

### Goal/Why
เพื่อเพิ่มความพึงพอใจและลดการโทรศัพท์ติดตามของญาติผู้ป่วยผ่านระบบ Automated LINE Notification และช่วยให้ผู้บริหารมองเห็นภาพรวมทรัพยากร รวมถึงพื้นที่ที่มีความต้องการสูง (Heat Map) เพื่อวางแผนงบประมาณจัดซื้อได้อย่างถูกต้อง

### User Stories

**User Story 1**
- As a ญาติผู้ป่วย
- I want to ได้รับข้อความแจ้งเตือนผ่าน LINE ทุกครั้งที่สถานะคำร้องเปลี่ยน (เช่น อนุมัติแล้ว, กำลังจัดส่ง, หรือใกล้ครบกำหนดคืน)
- So that ฉันสามารถเตรียมตัวสแตนบายรอรับอุปกรณ์ที่บ้านได้โดยไม่ต้องโทรสอบถามเจ้าหน้าที่

**User Story 2**
- As a ผู้บริหาร / เจ้าหน้าที่ระดับสูง
- I want to ดู Dashboard สรุปยอด และแผนที่ความหนาแน่นของผู้ป่วย (Heat Map)
- So that ฉันสามารถวิเคราะห์แนวโน้มการใช้งานและวางแผนจัดซื้ออุปกรณ์เพิ่มเติมในจุดที่ขาดแคลนได้

### Functional Requirements

**LINE Messaging API Integration**
- ระบบต้อง Trigger ข้อความอัตโนมัติไปยัง LINE ของญาติผู้ป่วย (ตามเบอร์โทร/LINE ID ที่ผูกไว้)
- เมื่อสถานะใน Workflow เปลี่ยนแปลง (มีทั้งหมด 7 Triggers ตามเอกสาร PRD)

**Dashboard Aggregation Metrics**
- หน้า Dashboard ต้องดึงข้อมูลมาคำนวณและแสดงผลตัวเลขสำคัญ 5 ค่าแบบ Real-time/Near Real-time:
  - จำนวนผู้ป่วยทั้งหมด (Total Patients)
  - จำนวนการยืมปัจจุบัน (Active Loans)
  - จำนวนอุปกรณ์คงเหลือแยกตามประเภท (Available Inventory)
  - จำนวนอุปกรณ์ที่ถูกยืมอยู่ (On-Loan Inventory)
  - จำนวนอุปกรณ์ชำรุด (Damaged Inventory)

**Geographic Heat Map Viewer**
- นำค่าพิกัด Latitude และ Longitude ของผู้ป่วยทุกคนที่มีสถานะ Active มา Render บน Component Map
- แสดงเป็น Heat Map ความหนาแน่นเชิงพื้นที่

### Acceptance Criteria

**Scenario 1: สถานะคำร้องเปลี่ยน ระบบส่ง LINE อัตโนมัติ**
- Given: ระบบบันทึกเบอร์โทรศัพท์/ช่องทางติดต่อ LINE ของญาติผู้ป่วยไว้เรียบร้อยแล้ว
- When: เจ้าหน้าที่กดเปลี่ยนสถานะใบคำร้องจาก "เตรียมจัดส่ง" เป็น "จัดส่งสำเร็จ"
- Then: LINE Webhook ต้องทำงานทันที และญาติผู้ป่วยต้องได้รับข้อความแจ้งเตือนว่า "อุปกรณ์จัดส่งสำเร็จเรียบร้อยแล้ว" ภายในเวลา < 5 วินาที

**Scenario 2: การแสดงผล Heat Map**
- Given: มีผู้ป่วยในระบบปักหมุดอยู่ในเขต "อำเภอเมือง" จำนวน 50 ราย และ "อำเภอหางดง" จำนวน 5 ราย
- When: ผู้บริหารเปิดหน้า Dashboard Analytics ดูในส่วน Geo-Location Map
- Then: ตัวแผนที่ต้องแสดงแถบสีความร้อน (Heat Map) สีแดงเข้ม/หนาแน่นในพื้นที่เขต "อำเภอเมือง" และสีจางในเขต "อำเภอหางดง" อย่างถูกต้องสัมพันธ์กับ Data ในระบบ

### Key Constraints/Non-Goals

**Key Constraint**
- ระบบ LINE Notification ใน Phase 1 จะเป็นการส่งแบบ One-way Broadcast/Push Message เท่านั้น
- ยังไม่รองรับระบบ Chatbot ถาม-ตอบอัตโนมัติแบบสองทาง (Two-way interaction)

**Non-Goal สำหรับ Phase 1**
- Dashboard ในส่วนนี้จะเป็นการสรุปข้อมูลอดีตจนถึงปัจจุบัน (Historical & Current Data)
- ยังไม่มีระบบ AI Predictive Dashboard (การคาดการณ์อนาคต) ซึ่งถูกจัดอยู่ใน Phase 2 Roadmap
